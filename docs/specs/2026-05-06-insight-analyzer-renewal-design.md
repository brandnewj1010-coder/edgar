## 0. TL;DR — 한 장 요약

> 지금의 InsightAnalyzer는 **"AI가 한 번에 만들어주는 마크다운 보고서"** 형태다. 형식적으로 깔끔하지만,
> 사용자가 보고 “**우와**” 할 만한 시각적 임팩트와 인사이트의 깊이가 부족하다.
> 핵심 원인은 ① **데이터를 직접 안 가져오고 검색에 의존**, ② **시각화는 Sankey 1종**, ③ **재무제표 표가 단조**(수치 정렬·포맷·음수색 모두 미적용), ④ **인사이트 카피가 표층적**(프롬프트가 "간결함" 강조)이다.
>
> 리뉴얼의 핵심 방향은 다음 세 줄로 요약된다:
>
> 1. **검색 기반 → 데이터(공시) 기반**으로 신뢰축을 옮긴다 (DART OpenAPI / EDGAR XBRL 직연동).
> 2. **마크다운 한 덩어리 → 모듈형 인사이트 카드**로 화면을 쪼갠다 (KPI·트렌드·이슈·비교 카드).
> 3. **AI = 요약기**가 아니라 **AI = 해설가/큐레이터**로 역할을 재정의한다 (한 줄 비유, 시나리오 분석, 이상치 자동 코멘트).

이 세 축이 곧 우리의 **wow포인트** 후보군이다. 자세한 내용은 §5~§9.

## 3. 핵심 문제 진단

사용자의 두 가지 핵심 불만:
1. “재무제표가 깔끔하지 않다.”
2. “인사이트가 약하고 wow포인트가 없다.”

코드를 근거로 5가지 원인으로 분해한다.

### 3.1 [P1] 표가 “문서”다 — 인터랙티브 데이터가 아님

**증상**
- 마크다운 표가 글자 그대로 렌더링되어 정렬·포맷이 일관성이 없다.
- `index.css:154` `text-left` 기본값 → **수치도 좌측 정렬**.
- 천 단위 콤마가 모델 출력에 의존 → `1000000` 그대로 들어오기도 함.
- 음수 빨강, ▲▼ 트렌드 마커, 셀 막대형 미니 게이지 모두 없음.
- 표 헤더와 본문 구분이 약함(`from-slate-50 to-slate-50/50`만으로는 약함).

**근거 위치**
- `src/index.css:147-180`
- `src/components/MarkdownReport.tsx:67-96` — 표를 그냥 마크다운으로 받아 그림.

**임팩트**
사용자가 “**숫자를 한눈에 비교**”하려고 들어온 화면에서 가장 중요한 자산(=숫자)이 가장 약하게 보인다. 이것이 “깔끔하지 않다”의 가장 큰 원인.

### 3.2 [P1] 시각화가 한 종류뿐, 그것마저도 “추정” 라벨

**증상**
- 차트는 Sankey 1종.
- Sankey 자체 품질도 약점:
  - `SankeyViz.tsx:67-71` 노드 툴팁이 **label만** 보여주고 값은 미표시 (link 툴팁만 값 표시).
  - 라벨에 천단위·단위 포맷팅 없음 (Nivo가 그대로 표시).
  - `App.tsx:253-256` 사용자에게 “**리포트·검색으로 추정한 구조입니다. 공시 원문과 다를 수 있습니다.**”라는 자조적 문구를 노출. → 신뢰도를 스스로 깎고 있음.
- KPI 카드, 시계열 라인, YoY 막대, 부문별 트리맵, 워터폴 차트, 게이지(부채비율 등) **전부 없음**.

**임팩트**
"보고서를 시각으로 한 번에 이해" 라는 가장 큰 wow 발생 지점이 비어 있다.

### 3.3 [P1] 데이터 신뢰축이 약함 — DART/EDGAR 직연동이 없음

**증상**
- `api/analyze.ts:18-80` 프롬프트는 “**원문 PDF 전체를 직접 읽을 수는 없을 수 있으므로**” 라고 명시. → 검색 결과 + 일반 지식으로 작성.
- 결과적으로 "네이버 뉴스를 잘 정리한 글" 수준이 될 수 있음.
- “이 숫자 어디서 왔어요?” 라는 질문에 답할 수 없음 (Grounding 출처는 있지만 “네이버 블로그”인 경우 多).

**임팩트**
교육·학습 도구라 해도 “**근거 있는 숫자**” 를 제공해야 인사이트의 무게가 생긴다. 지금은 “읽는 재미는 있지만 인용하긴 망설여지는” 콘텐츠.

### 3.4 [P2] 프롬프트가 “간결” 위주로 짜여 있어 인사이트 깊이 부족

**증상**
- `api/analyze.ts:50` `**간결하게** 작성하세요(각 섹션 과도한 장문은 피하고, 핵심 위주).`
- 출력 섹션이 고정 5개(개요/핵심/현금흐름/리스크/용어).
- 비교(compareWith) 시에도 "표로 정리, 마지막 한 줄 요약"만 요구. 시나리오·동종업계 벤치마크·전략적 시사점 미요청.
- “주목할 한 줄” 같은 **헤드라인 인사이트**가 없음.

**임팩트**
모델은 결국 “**시키는 만큼만**” 한다. 지금 프롬프트는 깊은 사유를 시키지 않고 있다.

### 3.5 [P2] 정보 구조가 “보고서를 읽게 만든다” — 인사이트 우선이 아님

**증상**
- 메인 영역이 `mx-auto max-w-3xl` → **데스크톱 와이드 화면에서 좌우 여백이 거대**(`App.tsx:204`).
- 위에서 아래로 “메타 → 액션바 → KRW어시스트 → Sankey → 마크다운”의 일자형 스크롤. 핵심 숫자가 위로 안 올라옴.
- 학습 패널(LearningSide)이 22rem 고정인데 정작 거기에 들어가는 “용어·출처” 가 콘텐츠 없을 때 빈약하게 보임.
- 리포트가 길어질수록 **"어디부터 봐야 하는지" 가이드가 사라짐**.

**임팩트**
스크롤이 길어 처음의 “와” 가 사라짐. 첫 1스크린에 wow가 와야 한다.

## 4. 타겟 재정의 — 옵션 A / B / C

지금은 누가 쓰는지 약간 흐릿하다 (`api/analyze.ts:62` "한국 대학생용 교육 톤"이라는 단서만 있음). 리뉴얼은 타겟을 명확히 한 발 더 좁혀야 wow포인트가 또렷해진다.

| 옵션 | 타겟 | 핵심 가치 제안 | 적합한 wow |
|------|------|----------------|-----------|
| **A. 학습형** | 회계/재무 입문자, 대학생·취준생, 자격증 학생 | "공시를 보며 회계를 배운다" | 용어 팝오버 강화, 학습 트랙·연속 학습일, 미션·뱃지 |
| **B. 의사결정형** | 주니어 직장인 투자자, 자영업자 | "5분 만에 한 회사를 머릿속에 정리한다" | 한 줄 헤드라인, KPI 카드, 동종업계 자동 비교, 리스크 신호등 |
| **C. 실무 보조형** | 현직 IR/리서치/스타트업 기획자 | "공시 초벌 분석을 1시간 → 5분으로" | 출처 인용 카드, MD/PDF/PPT 익스포트, 변경 감지 알림 |

> **추천: B를 메인, A를 보조**.
> 이유는 ①현재 도메인(`edgar-roan.vercel.app`)·메타("공시로 회계 공부")가 학습 쪽에 가깝지만, ②사용자 핵심 불만(“wow가 없다”)을 해결하려면 **시각·인사이트 임팩트가 있는 의사결정형** 요소를 메인에 끌어와야 하기 때문. ③기존 학습 자산(용어 411개·퀴즈·성찰)은 사이드로 밀어 “보너스”로 두고, 메인은 ‘한 회사 5분 이해’ 카피로 다시 잡는다.

이 추천을 받아들이는 경우의 새 카피 제안:

> **새 슬로건 후보**
> - "공시 한 번에. 회사 한눈에."
> - "5분 만에 보는 한국·미국 공시 인사이트."
> - "Read less. Understand more — DART·EDGAR AI 인사이트."

## 5. 리뉴얼 큰 방향 3택

### 5.1 옵션 1 — 가벼운 리프레시 (1~2주)
- 표 가독성 5종 패치 (수치 우측정렬, 천단위, 음수 빨강, YoY 마커, 헤더 강조)
- Sankey 노드 툴팁 값 표시 + 라벨 포맷
- 헤드라인 카피 1줄 AI 추가 (프롬프트만 수정)
- 마크다운 표 → JSON 추출해 KPI 카드 1단(매출/이익/이익률) 자동 생성
- max-w-3xl 해제, 와이드 그리드로 변환

**장점:** 코드 변경 최소, ROI 즉각.
**단점:** 데이터 신뢰성·시각화 다양성 같은 근본 약점은 여전.

### 5.2 옵션 2 — 중간 리뉴얼 (4~6주) ★ 추천
- 옵션 1 전체 +
- DART OpenAPI 직연동 (재무제표 표준 수치 캐싱)
- KPI 카드 4종 + YoY 라인 + 비교 막대 + 부문별 트리맵 (총 5종 시각화)
- AI 역할 재설계: “요약” 대신 “해설/이상치 코멘트/시나리오”
- IA 개편: “인사이트 우선, 마크다운은 접힘”
- 인증(소셜 로그인) + RLS 정상화 + 사용량 미터링

**장점:** 사용자가 곧장 “리뉴얼했네!”를 인지. 데이터 신뢰성·시각화·인사이트 3축 모두 개선.
**단점:** DART API 의존, 캐싱·정규화 작업 필요.

### 5.3 옵션 3 — 풀 리뉴얼 (8~12주)
- 옵션 2 전체 +
- EDGAR XBRL 직접 수집 + 재무제표 정형화
- 산업 분류(KSIC/SIC) 기반 자동 동종업계 비교
- 변경 감지(공시 RSS) + 알림
- 리포트 익스포트(PPT/PDF) + 공유 링크
- 모바일 PWA + 다크모드

**장점:** 완전한 제품. 유료화 / B2B 시도 가능.
**단점:** 큰 비용, 백엔드 인프라 부담.

|------|------|------|
| 1차 | Google Search Grounding (검색 텍스트) | (유지) — 보조용 |
| 2차 | (없음) | **DART OpenAPI** (`opendart.fss.or.kr`) — 단일회사 주요계정 / 다중회사 주요계정 |
| 3차 | (없음) | **EDGAR XBRL** (`data.sec.gov/api/xbrl/companyfacts/CIK*.json`) |
| 4차 | (없음) | 캐시(Supabase 또는 KV) — 동일 종목·연도 조회는 1회만 외부호출 |

### 6.2 새 데이터 흐름 제안

```
사용자 입력 (예: 005930, 2024)
     │
     ▼
[캐시 확인] ── hit ──▶ DB의 표준화 재무지표
     │ miss
     ▼
[DART OpenAPI] 단일회사 주요계정 → 매출/영업이익/순이익/EPS/부채/자본 등 표준 수치
     │
     ▼
[정규화 / 단위 통일 / YoY 계산]
     │
     ▼
[Gemini 호출] → 해설·이상치·시나리오·헤드라인 (숫자는 시스템이 책임)
     │
     ▼
[프론트] KPI 카드 + 차트 + AI 해설 카드 (각각 출처 칩 부착)
```

**핵심:** “**숫자는 코드가, 해석은 AI가**” 책임 분리. 지금은 둘 다 AI가 떠안고 있어 신뢰가 흔들린다.

### 6.3 EDGAR(미국) 측 보강

EDGAR는 XBRL 표준이 매우 잘 갖춰져 있어 직접 활용이 사실상 강제다.
- `companyfacts/CIK{10자리}.json` 한 번에 수년치 수치 회수.
- 필요 시 SEC 동시 요청 한도(초당 10건) 준수, User-Agent 필수.

### 6.4 비용·법무 체크 리스트 (오픈 이슈)
- DART API 사용 신청 / 사용량 한도 / 인증키 보관 정책.
- Gemini 토큰 비용: 단계별 호출(2단계 → 3단계: 헤드라인, 해설, 시나리오)로 늘면 토큰 증가.
- “투자 권유 아님” 디스클레이머 위치(현재는 프롬프트에만 있음 → UI에 상시 노출 필요).

### 8.10 AI 해설 카드 — 프롬프트 재설계

## 9. 시각·디자인 디테일

지금 디자인 톤은 인디고·세리프 기반의 **"교양 잡지 같은 분위기"** 라 좋다. 그대로 가면서 다음만 보강:

- **숫자는 mono 또는 tabular-nums 강제** — `font-variant-numeric: tabular-nums;`를 KPI/표/차트 라벨에 일괄 적용.
- **YoY 화살표** — 색·기호 통일 (▲▼ 또는 lucide의 ArrowUpRight/Down).
- **Sankey/Waterfall 색 토큰** — `revenue=indigo-600`, `profit=emerald-600`, `expense=rose-600`, `neutral=slate-500` 로 디자인 토큰화. (현재는 SankeyViz에서만 별도 색을 쓰고 있음 → 토큰화)
- **카드 그림자 통일** — KPI 카드는 미세 그림자 + hover 시 ring-indigo 보더.
- **세리프 = 본문 / 산세리프 = 데이터** 원칙. KPI 숫자는 산세리프 + tabular-nums로 명료하게.
- **다크모드** — Phase 3 후순위. 옵션 2 단계까지는 라이트만.

## 13. 관찰 노트 (코드를 읽다 발견한 것들)

기획과 직접 관련은 작지만 의사결정에 도움이 될 메모.

1. **`search_history` 테이블이 정의돼 있지만 안 쓴다.** (`supabase/migrations/20260401000000_search_history.sql`) — 향후 검색 트렌드를 보여주거나 자동완성에 활용 가능.
2. **`analysis_reports` RLS가 `using (true)` / `with check (true)` 로 사실상 무방비.** 인증 도입 시 `user_id` 컬럼 추가 + `auth.uid()` 기반 정책 필수.
3. **`.env.example`에 실제 키처럼 보이는 문자열이 있다(`GEMINI_API_KEY=AIzaSy…`).** Phase 0 PR에서 함께 마스킹/문구 수정 권장.
4. **`fiscalYears` 옵션이 2023/2024/2025 하드코딩.** 동적으로 “최근 5년” 등으로 바꿀 수 있음.
5. **`.tsconfig.api.json`이 따로 있어 함수 빌드 분리가 잘 돼 있다.** 외부 데이터 소스 추가가 비교적 쉬움.
6. **`LearningSide`는 `lg:hidden` + `lg:block` 두 곳에서 두 번 렌더된다.** 모바일·데스크톱 동기화에 유의.
7. **`KrwAssistView`의 환율 fallback이 잘 짜여 있어** 새 환율 사용처에 그대로 재사용 가능.
8. **`TermSpan`의 매칭 알고리즘(긴 단어 우선·CJK 처리)이 의외로 정교**하다 — 학습 모드의 핵심 자산이므로 보존.
